## LLM议会是什么

**LLM议会（LLM Council）** 是一个多模型协作回答系统。用户提出问题后，多个LLM模型先独立作答，再互相匿名评审，最后由一个"主席模型"综合各方意见，给出最终答案。

简单说，就是**让AI自己开会，自己判断哪个答案更好**。

## 为什么值得关注

传统的AI使用模式是"一问一答"：你问GPT，它答。但你不知道这个答案和Claude比谁更好，和Gemini比谁更准。

LLM Council 把这个过程变成了一个**多模型竞争会议**。结果不只是"答案"，而是"哪个答案更好+为什么"。

## 3步拆解：议会是怎么开的

整个流程分三个阶段，技术细节全在这：

**第一步：初评阶段**
所有配置的模型（GPT-5.1、Gemini 3 Pro、Claude Sonnet 4.5、Grok等）**同时独立生成答案**。用户看不到模型身份，只看到编号（Model A、Model B……）。

这一步的目的是让每个模型充分表达自己的理解和推理风格。

**第二步：互审阶段**
每个模型的答案被匿名分发给其他所有模型。模型们在**不知道对方是谁**的情况下，给每份答案打分，并写出简短评审意见。

匿名设计是为了**消除偏见**——如果知道是GPT评Claude，可能会有不同的期待。

**第三步：终审阶段**
指定的"主席模型"（Karpathy默认用Gemini 3 Pro）读取所有原始答案和评审意见，综合权衡后，给出最终结论。

最终答案不仅包含结论本身，还包含评审过程的摘要。

## 我复现了一套，效果怎么样

搭建环境需要三步：

**第一步：安装依赖**
后端用 `uv sync`，前端进 `frontend/` 跑 `npm install`。需要准备一个 OpenRouter API Key（用于调用多模型）。

**第二步：配置模型**
在 `backend/config.py` 里定义 `COUNCIL_MODELS`（参会议员模型）和 `CHAIRMAN_MODEL`（主席模型）。可以按预算和场景灵活调整组合。

**第三步：启动服务**
执行 `./start.sh`，或手动分别启动后端（`uv run python -m backend.main`）和前端（`npm run dev`）。

访问 `localhost:5173`，输入问题，议会就开始了。

## 适合谁用

LLM Council 不是一个普通聊天机器人，它是一个**模型评测与协作框架**。这几个场景特别适合：

**技术调研**：让不同模型分析同一段代码，对比它们的理解差异。
**写作优化**：多模型对同一主题给出不同视角，综合出更全面的答案。
**教学备课**：一个问题，多个模型从不同专业角度回答，找出最佳解释路径。

## 局限性也要说清楚

项目是 Karpathy 的"周六Hack"项目，不是生产级应用。

最直接的问题是**成本**——每次提问要同时调多个模型，按token计费。跑一次议会大概消耗单次对话5-10倍的token。

另外，匿名互审机制虽然消除了身份偏见，但**模型之间的能力差异仍然会影响评分公正性**——更强的模型在评审时天然更有说服力。

## Q&A：常见疑问

**Q：没有OpenRouter能用吗？**
A：不能。LLM Council 依赖 OpenRouter 统一接入多个模型。如果只用单一模型，这个框架的价值就消失了。

**Q：主席模型能不能换？**
A：可以。在配置里改 `CHAIRMAN_MODEL` 就行。Karpathy 用 Gemini 3 Pro，也可以换成 Claude 或 GPT-4。

**Q：我的电脑跑得动吗？**
A：模型都在云端跑，你的电脑只负责运行前端界面。配置要求极低，普通的笔记本都行。
